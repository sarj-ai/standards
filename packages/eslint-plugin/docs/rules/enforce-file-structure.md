# `enforce-file-structure`

> A TypeScript file must lay out its top-level statements in a fixed order: imports → types → constants → functions → exports. Server-action files must start with the `'use server'` directive.

## What it catches

- Top-level statements out of canonical order (e.g. a `const` before any `import` it depends on, or a `type` defined after the function that uses it).
- A file under `app/**/actions.ts` (or any module with default-exported async functions used as server actions) that's missing the `'use server'` directive.

## Why we encourage the alternative

Files written top-down read like a story:

1. **What dependencies do I need?** (`import` statements at the top)
2. **What shapes do I work with?** (type aliases, `interface`, Zod schemas)
3. **What constants and config are at the top of the module?**
4. **What functions implement the behavior?**
5. **What's exported to consumers?** (`export { ... }` at the bottom, mirroring Go's idiom)

Reviewers don't have to chase declarations up and down. CI catches the drift early — `eslint --fix` even auto-corrects most violations.

The `'use server'` requirement is a Next.js correctness issue: forget the directive on a Server Action module and the function gets bundled into the client, leaking server-only logic.

## Bad

```ts
const MAX_RETRIES = 5; // constant before imports

import { client } from "./client";

async function fetchUser(id: string) { /* ... */ }

type User = { id: string; name: string }; // type after the function that uses it

export { fetchUser };
```

## Good

```ts
import { client } from "./client";

type User = { id: string; name: string };

const MAX_RETRIES = 5;

async function fetchUser(id: string): Promise<User> {
  /* ... */
}

export { fetchUser };
```

## More examples

**Server action file**:

```ts
// Bad — directive missing
import { db } from "@/db";
export async function createOrder(form: FormData) { /* ... */ }

// Good
"use server";

import { db } from "@/db";
import { z } from "zod";

const ZOrder = z.object({ items: z.array(z.string()) });

export async function createOrder(form: FormData) { /* ... */ }
```

**Type-only re-exports** — keep them with other types, not at the bottom:

```ts
// Good
import type { User } from "./types";

export type { User };

type LocalAlias = User & { sessionId: string };

const DEFAULT_TIMEOUT_MS = 5_000;
// ...
```

## When to suppress

Generated code (codegen output, `next.config.js`, etc.) often violates the ordering by design. Suppress per-file at the top of the file:

```ts
/* eslint-disable @sarj/enforce-file-structure -- generated by openapi-codegen */
```
