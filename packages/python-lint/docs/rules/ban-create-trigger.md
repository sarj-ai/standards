# `sarj-ban-create-trigger`

> Don't put business logic in database triggers. Encode the same rules in the application code where they're version-controlled, code-reviewed, and observable.

## What it catches

A regex match for `(?i)CREATE TRIGGER` in any file matching the consumer's `files:` glob (typically Postgres migrations).

## Why we encourage the alternative

Triggers have a tempting promise: "if the data changes, this side-effect always runs, no matter who's writing." In practice, that becomes a maintenance hazard:

1. **Invisible to the application.** Trigger logic doesn't show up in `git blame` for the service that's writing. New engineers debugging a write that mysteriously produces three audit rows have to know to go look at the DDL.
2. **Hard to test.** Most service tests use fakes or in-memory stores; triggers only exist in a real Postgres. The trigger logic is either untested or duplicated in test fixtures.
3. **Hard to evolve.** Changing trigger logic requires a migration with downtime risk; the same change in application code is an atomic deploy.
4. **Observability gap.** OpenTelemetry/Sentry/Langfuse instrumentation lives in the application code path. Triggers fire silently — no spans, no log lines.
5. **Concurrency footguns.** A trigger that updates another row introduces lock-ordering complexity that the application can't see.

Almost every legitimate use of a trigger can be expressed cleanly in service code. The few cases where a trigger is actually correct (point-in-time audit at the database level for compliance, vendor-supplied replication helpers) are rare and worth explicit team discussion.

## Bad

```sql
CREATE TRIGGER audit_orders_after_update
AFTER UPDATE ON orders
FOR EACH ROW
EXECUTE FUNCTION audit_log_insert();
```

## Good

```py
# In the service that writes orders:
async def update_order(order_id: str, patch: OrderPatch) -> Order:
    async with db.transaction():
        order = await order_store.update(order_id, patch)
        await audit_log_store.insert(
            AuditEntry(
                actor=current_actor(),
                entity_type="order",
                entity_id=order_id,
                action="update",
                diff=patch.model_dump(),
            )
        )
        return order
```

Now the audit insert is:
- Tested in the service's unit + integration tests
- Visible in a trace/span
- Skipped by mistake? `git blame` shows it
- Wrong field name? Static typecheck catches it

## More examples

**`updated_at` auto-update trigger** — yes, this is the canonical trigger example. Replace with an application-set timestamp:

```py
# Good
await order_store.update(order_id, patch.model_copy(update={"updated_at": datetime.now(UTC)}))
```

…or use `DEFAULT now()` + `ON UPDATE` via a column generated by a clean DEFAULT/UPDATE pattern, but most teams just set it in code.

**Soft-delete cascade** — express it as an explicit application-level cascade:

```py
async def soft_delete_user(user_id: str) -> None:
    async with db.transaction():
        await user_store.soft_delete(user_id)
        await session_store.revoke_for_user(user_id)
        await api_key_store.deactivate_for_user(user_id)
```

vs a trigger that cascades on `users.deleted_at IS NOT NULL`.

## When to suppress

The narrow legitimate cases (point-in-time audit for compliance, vendor replication, materialized-view refresh hooks) — exclude the specific migration file in the consumer's `.pre-commit-config.yaml`:

```yaml
- id: sarj-ban-create-trigger
  files: '^svcs/db/db/migrations/.*\.sql$'
  exclude: |
    (?x)^(
      svcs/db/db/migrations/20240601_compliance_audit_trigger\.sql$
    )
```

Document the rationale in the migration file itself (a `-- ...` comment explaining why this is one of the rare legit triggers).
